# Command Template Reference

Complete template and examples for generating slash commands.

## Standard Command Structure

```markdown
---
name: command-name
description: "Brief description with workflow graph summary"
category: workflow|automation|analysis|generation|deployment|testing|maintenance|orchestration
complexity: simple|standard|complex|high
mcp-servers: [list, of, mcp, servers]
personas: [list, of, expert, personas]
---

# /command-name - Human Readable Title

> **Workflow Description**: Detailed explanation of what this workflow does

**Detected Patterns**: pattern1, pattern2, pattern3

## Workflow Graph

\`\`\`
[Graph visualization using symbols - see WGDL standard below]
\`\`\`

**IMPORTANT**: All workflow graphs must follow the **Workflow Graph Description Language (WGDL)** standard defined in `graph-description-language.md`. Key requirements:

- Use standardized symbols: `→` (sequential), `⎡⎢⎣` (parallel), `├─└─` (branch), `⚡` (async), `↻` (loop), `📦` (shared)
- Include unique node IDs: `[node_X]`
- Specify dependencies explicitly: `(depends on: node_1, node_2)`
- Follow indentation rules for readability
- Validate against 6 WGDL validation rules before generation

**See**: `docs/graph-description-language.md` for complete WGDL specification, grammar, and examples.

## Behavioral Flow

1. **Step Name**
   - Description of step
   - Expected behavior

2. **Next Step** (parallel|conditional|async)
   - Description

## MCP Integration

- **server-name MCP**: Rationale for using this server

## Usage Examples

\`\`\`bash
/command-name

# Expected behavior description
\`\`\`

## Boundaries

**Will:**
- What the command will do

**Will Not:**
- What the command won't do

---
*Generated by workflow-creator skill on DATE*
```

## Metadata Fields

### name (required)
- **Format**: lowercase-with-hyphens
- **Length**: 3-30 characters
- **Pattern**: `^[a-z][a-z0-9-]*$`
- **Examples**:
  - `deploy-with-tests`
  - `batch-type-annotate`
  - `parallel-quality-check`

### description (required)
- **Length**: 50-200 characters
- **Content**: Brief workflow summary + graph snippet
- **Format**: "Action description. Graph: [first 3 nodes]..."
- **Examples**:
  - "Deploy to staging, verify health, conditional production deployment. Graph: → Staging → Verify ├─ [Pass] → Prod"
  - "Batch process files with parallel execution. Graph: → Batch[N files] ⎡→ Process"

### category (required)
- **workflow**: General multi-step processes
- **automation**: Repetitive/retry tasks
- **analysis**: Investigation workflows
- **generation**: Creation tasks
- **deployment**: Release pipelines
- **testing**: QA workflows
- **maintenance**: Cleanup/updates
- **orchestration**: Multi-workflow coordination

### complexity (required)
- **simple**: 1-3 steps, no special patterns (score < 0.3)
- **standard**: 4-8 steps, 1-2 patterns (score 0.3-0.6)
- **complex**: 9-15 steps, 2-3 patterns (score 0.6-0.85)
- **high**: 16+ steps, 3+ patterns (score >= 0.85)

### mcp-servers (optional)
- **Format**: YAML list `[server1, server2]`
- **Options**: `sequential`, `magic`, `context7`, `playwright`, `serena`
- **Empty**: `[]` for simple workflows
- **Selection**:
  - UI tasks → `magic` + `context7`
  - Testing → `playwright`
  - Complex → `sequential`
  - Code ops → `serena`

### personas (optional)
- **Format**: YAML list `[persona1, persona2]`
- **Limit**: 1-7 personas
- **Options**: See persona reference below
- **Selection**: Based on domain and tasks

## Example Templates

### 1. Simple Node Workflow

```yaml
---
name: check-config
description: "Validate system configuration file syntax and required fields"
category: analysis
complexity: simple
mcp-servers: []
personas: [general-purpose]
---

# /check-config - Configuration Validation

> **Workflow Description**: Single-step configuration file validation

**Detected Patterns**: node

## Workflow Graph

\`\`\`
→ ValidateConfig [node_1]
\`\`\`

## Behavioral Flow

1. **Validate Configuration**
   - Load configuration file
   - Check syntax (JSON/YAML)
   - Verify required fields
   - Report validation results

## Usage Examples

\`\`\`bash
/check-config

# Validates ./config.yaml or ./config.json
\`\`\`
```

### 2. Parallel Execution

```yaml
---
name: parallel-quality-check
description: "Run linter, tests, and type checker in parallel for maximum speed. Graph: ⎡→ Lint ⎢→ Test ⎣→ TypeCheck"
category: testing
complexity: standard
mcp-servers: [playwright]
personas: [qa-specialist, backend-developer]
---

# /parallel-quality-check - Parallel Quality Assurance

> **Workflow Description**: Execute multiple quality checks simultaneously

**Detected Patterns**: parallel

## Workflow Graph

\`\`\`
⎡→ RunLinter [task_1]
⎢→ RunTests [task_2]
⎣→ TypeCheck [task_3]
    ↓
→ AggregateResults [task_4]
\`\`\`

## Behavioral Flow

1. **Parallel Execution**
   - Run linter on all source files
   - Execute test suite
   - Perform type checking
   - All three run simultaneously

2. **Aggregate Results**
   - Collect all results
   - Report combined status
   - Fail if any check fails

## MCP Integration

- **playwright MCP**: Automated test execution and validation

## Usage Examples

\`\`\`bash
/parallel-quality-check

# Runs 3 checks in parallel, reports in ~30s instead of ~90s
\`\`\`
```

### 3. Complex Conditional Flow

```yaml
---
name: deploy-with-verification
description: "Deploy to staging, verify, conditional prod deployment. Graph: → Staging → Verify ├─ [Pass] → Prod └─ [Fail] → Rollback"
category: deployment
complexity: complex
mcp-servers: [sequential, playwright]
personas: [devops-architect, system-architect, security-engineer, qa-specialist]
---

# /deploy-with-verification - Verified Deployment Pipeline

> **Workflow Description**: Staged deployment with health verification and rollback

**Detected Patterns**: flow, branch, looping

## Workflow Graph

\`\`\`
→ DeployStaging [step_1]
→ HealthCheck [step_2]
  ↻ Retry up to 3 times (5s wait)
  ├─ [Healthy] → DeployProduction [step_3]
  │  → FinalVerify [step_4]
  │    ├─ [Success] → Complete [step_5]
  │    └─ [Fail] → RollbackProd [step_6]
  └─ [Unhealthy] → RollbackStaging [step_7]
\`\`\`

## Behavioral Flow

1. **Deploy to Staging**
   - Build and package application
   - Deploy to staging environment
   - Wait for deployment to stabilize

2. **Health Verification** (with retry)
   - Check application health endpoint
   - Retry up to 3 times if unhealthy
   - 5-second wait between attempts

3. **Conditional Production Deployment**
   - If healthy: Deploy to production
   - If unhealthy: Rollback staging

4. **Final Verification**
   - Verify production deployment
   - If fails: Automatic rollback

## MCP Integration

- **sequential MCP**: Complex decision flow coordination
- **playwright MCP**: Automated health check verification

## Usage Examples

\`\`\`bash
/deploy-with-verification

# Safe deployment with automatic rollback on failure
\`\`\`
```

## MCP Server Selection Guide

| Task Type | Primary | Secondary | Rationale |
|-----------|---------|-----------|-----------|
| UI Components | magic | context7 | Magic generates, Context7 provides docs |
| Testing | playwright | - | Browser automation |
| Complex Analysis | sequential | serena | Sequential coordinates, Serena for code |
| Code Operations | serena | sequential | Serena for symbols, Sequential for planning |
| Documentation | context7 | - | Official framework docs |

### Pattern: Batch Processing
```yaml
category: automation
complexity: standard
mcp-servers: []
personas: [general-purpose]
```

### Pattern: Parallel + Testing
```yaml
category: testing
complexity: standard
mcp-servers: [playwright]

```

### Pattern: Complex Deployment
```yaml
category: deployment
complexity: complex
mcp-servers: [sequential, playwright, serena]
personas: [devops-architect, system-architect, security-engineer]
```

### Pattern: Nested Workflows
```yaml
category: orchestration
complexity: high
mcp-servers: [sequential, magic, playwright, serena]
personas: [system-architect, devops-architect, frontend-architect, backend-architect, qa-specialist]
```

## Validation Checklist

Before generating command:

- [ ] Name is lowercase-with-hyphens
- [ ] Description includes workflow graph hint
- [ ] Category matches workflow nature
- [ ] Complexity score calculated correctly
- [ ] MCP servers justify their selection
- [ ] Personas align with tasks
- [ ] Workflow graph uses correct symbols
- [ ] Behavioral steps are clear
- [ ] Examples demonstrate usage
- [ ] Boundaries define scope

## Output Location

Generated commands should be placed in:
```
~/.claude/commands/workflows/[command-name].md
```

Ensure the directory exists before writing.
