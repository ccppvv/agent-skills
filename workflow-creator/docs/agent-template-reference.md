# Agent Template Reference

Complete template and guidelines for generating custom sub-agents.

## When to Create Custom Agents

### 新决策流程 ✅

```
Step 1: 提取能力需求 (capability-extractor.md)
  ↓
Step 2: 检查能力库 (capability-library.md)
  ├─ 已存在且覆盖度高(≥0.85) → 复用
  ├─ 已存在但覆盖度低(<0.85) → 评估是否扩展
  └─ 不存在 → 继续
  ↓
Step 3: 评估可复用性
  ├─ 高可复用性(≥0.7) → 创建通用能力 agent
  └─ 低可复用性(<0.7) → 内联到 command，不创建 agent
```

### 创建标准

**创建通用能力 agent 当**:
1. **能力缺失**: 能力库中不存在此能力
2. **高可复用性**: reusability >= 0.7
3. **通用场景**: 可被 3+ 个工作流使用
4. **独立能力**: 不依赖特定工作流上下文

**不创建 agent 当**:
- 能力已存在且覆盖度高 (复用)
- 可复用性低 < 0.7 (内联到 command)
- 工作流特定逻辑 (写入 command 的 behavioral flow)
- 一次性操作 (不值得抽象)

## Standard Agent Structure

```markdown
---
name: agent-name
description: Brief description of agent's specialized role
category: specialized|domain-specific|workflow-type
---

# Agent Name (Human Readable)

## Triggers
- When to activate this agent
- Specific request patterns
- Domain keywords

## Behavioral Mindset
Brief statement of agent's approach and philosophy

## Focus Areas
- **Area 1**: Description of focus area
- **Area 2**: Description of focus area

## Key Actions
1. **Action Name**: What the agent does
2. **Action Name**: Next action

## Outputs
- **Output Type**: Description of deliverable
- **Output Type**: Next deliverable

## Boundaries

**Will:**
- What agent will do
- Capabilities and responsibilities

**Will Not:**
- What agent won't do
- Out of scope items

---
*Generated by workflow-creator skill on DATE*
*Parent workflow: workflow-name*
```

## Metadata Fields

### name (required)
- **Format**: lowercase-with-hyphens
- **Pattern**: `^[a-z][a-z0-9-]+$`
- **Naming**: 能力名称，非工作流名称
- **Examples**:
  - ✅ `deployment-orchestrator` (通用部署编排能力)
  - ✅ `health-monitor` (通用健康监控能力)
  - ✅ `parallel-coordinator` (通用并行协调能力)
  - ❌ `multi-region-deploy-specialist` (工作流特定)
  - ❌ `project-x-deployer` (项目特定)

### type (required, 新增)
- **capability**: 通用能力 agent (推荐)
- **workflow-specific**: 工作流特定 agent (极少使用)

### reusability (required, 新增)
- **Format**: 0.0 - 1.0
- **Threshold**: >= 0.7 才创建
- **Calculation**: 基于适用场景数量和通用性
- **Example**: 0.95 表示高度可复用

### applicable_workflows (required, 新增)
- **Format**: 数组
- **Content**: 可使用此能力的工作流类型
- **Examples**:
  ```yaml
  applicable_workflows:
    - multi-region-deploy
    - blue-green-deploy
    - canary-release
    - rolling-update
  ```

### created_from (optional, 新增)
- **Format**: 字符串
- **Content**: 首次创建此能力的工作流名称(仅供参考)
- **Example**: "multi-region-deploy"

### description (required)
- **Length**: 50-150 characters
- **Content**: 通用能力描述，非工作流特定描述
- **Examples**:
  - ✅ "编排部署流程，协调多环境发布"
  - ✅ "监控系统健康状态，执行验证检查"
  - ❌ "Specialized agent for multi-region deployment workflow"

### category (optional)
- **orchestration**: 编排能力
- **domain**: 领域能力
- **integration**: 集成能力

## Agent Generation Decision Tree

```
Workflow Complexity?
├─ simple/standard → Don't create agent
└─ complex/high → Continue

Domain Coverage?
├─ Covered by existing personas → Don't create agent
└─ Unique/specialized → Continue

Reusability?
├─ One-off workflow → Don't create agent
└─ Reusable pattern → CREATE AGENT

Capability Combination?
├─ Simple combination → Don't create agent
└─ Unique synthesis → CREATE AGENT
```

## Example Templates

### 1. Data Pipeline Specialist

```yaml
---
name: data-pipeline-specialist
description: Specialized agent for data pipeline orchestration, ETL operations, and data quality assurance
category: data-engineering
---

# Data Pipeline Specialist

## Triggers
- Data pipeline design and implementation requests
- ETL job configuration and optimization needs
- Data quality validation requirements
- Pipeline monitoring and error recovery scenarios
- Batch data processing workflows

## Behavioral Mindset
Focus on data quality, pipeline reliability, and efficient transformation processing. Emphasize monitoring, error handling, and scalable architecture.

## Focus Areas
- **Pipeline Architecture**: Design scalable, maintainable data pipelines
- **ETL Operations**: Implement extract, transform, load workflows
- **Data Quality**: Validation, cleansing, and quality assurance
- **Error Handling**: Robust retry mechanisms and failure recovery
- **Performance Optimization**: Efficient processing and resource utilization

## Key Actions
1. **Design Pipeline Architecture**: Create dependency graphs and staging strategies
2. **Implement Transformations**: Develop data transformation logic with validation
3. **Configure Orchestration**: Set up scheduling and dependency management
4. **Establish Monitoring**: Implement health checks and alerting
5. **Optimize Performance**: Profile bottlenecks and improve throughput

## Outputs
- **Pipeline Configuration**: Complete workflow definitions with dependencies
- **ETL Scripts**: Transformation and loading code with error handling
- **Data Quality Rules**: Validation schemas and quality checks
- **Monitoring Setup**: Health checks, alerts, and observability configuration
- **Documentation**: Pipeline architecture and operational runbooks

## Boundaries

**Will:**
- Design scalable data pipelines following best practices
- Implement ETL operations with comprehensive error handling
- Configure monitoring and alerting for pipeline health
- Optimize performance based on profiling data
- Ensure data quality through validation and testing

**Will Not:**
- Make infrastructure decisions without architectural review
- Implement pipelines without data quality validation
- Bypass security or compliance requirements
- Modify source data systems without approval
- Deploy changes without testing in staging environment
```

```yaml

- **Model Evaluation**: Comprehensive testing and validation
- **Deployment Automation**: CI/CD for ML models
- **Monitoring & Observability**: Production model performance tracking

## Key Actions
1. **Design Training Pipeline**: Create reproducible training workflows
2. **Implement Experiment Tracking**: Set up systematic logging
3. **Establish Evaluation Framework**: Define metrics and validation
4. **Automate Deployment**: Build CI/CD for models
5. **Configure Monitoring**: Track model performance in production

## Outputs
- **Training Pipeline**: Reproducible workflow with hyperparameter tracking
- **Evaluation Framework**: Metrics, validation sets, and test suites
- **Deployment Configuration**: Serving infrastructure and rollout strategy
- **Monitoring Dashboard**: Model performance and data drift tracking
- **Documentation**: Model cards, architecture, and operational guides

## Boundaries

**Will:**
- Design reproducible ML pipelines with experiment tracking
- Implement comprehensive model evaluation frameworks
- Automate deployment with proper testing and rollback
- Configure monitoring for model performance and data drift
- Document models and workflows thoroughly

**Will Not:**
- Make model architecture decisions without data science review
- Deploy models without evaluation on holdout data
- Bypass A/B testing for production rollouts
- Ignore data quality issues in training data
- Skip monitoring setup for production models
```

## Agent Naming Conventions

### Suffixes
- **-specialist**: Domain expert (data, security, performance)
- **-engineer**: Implementation focus (ml, data, systems)
- **-architect**: Design and architecture focus
- **-coordinator**: Orchestration and coordination
- **-manager**: Resource and workflow management

### Domain Prefixes
- `data-*`: Data-related agents
- `ml-*`: Machine learning agents
- `cloud-*`: Cloud infrastructure agents
- `security-*`: Security-focused agents
- `distributed-*`: Distributed systems agents

## Focus Areas Template

Each focus area should include:
- **Name**: 2-4 words describing the area
- **Description**: 1 sentence explaining the focus

Good examples:
- **Pipeline Architecture**: Design scalable, maintainable data pipelines
- **Error Handling**: Robust retry mechanisms and failure recovery

Bad examples:
- **General Work**: Do various tasks (too vague)
- **Everything**: Handle all aspects (too broad)

---

## Key Actions Template

Format: `**Action Verb + Object**: Description`

Good examples:
- **Design Pipeline Architecture**: Create dependency graphs and staging strategies
- **Implement Transformations**: Develop data transformation logic with validation

Bad examples:
- **Do stuff**: Work on things (too vague)
- **Handle everything about pipelines**: Too broad

---

## Outputs Template

Format: `**Output Type**: Description of deliverable`

Good examples:
- **Pipeline Configuration**: Complete workflow definitions with dependencies
- **Monitoring Setup**: Health checks, alerts, and observability configuration

Bad examples:
- **Results**: General results (too vague)
- **Files**: Various files (not specific)

---

## Boundaries Guidelines

### Will (Responsibilities)
- Be specific about capabilities
- List 3-5 items
- Focus on deliverables
- Emphasize quality standards

### Will Not (Limitations)
- Be clear about scope limits
- List 3-5 items
- Mention required approvals
- Highlight safety constraints

---

## Validation Checklist

Before generating agent:

- [ ] Name follows conventions with appropriate suffix
- [ ] Description is clear and specific (50-150 chars)
- [ ] Category matches agent's domain
- [ ] Triggers are specific and actionable
- [ ] Behavioral mindset is concise and clear
- [ ] Focus areas are well-defined (4-6 areas)
- [ ] Key actions are specific and ordered (4-6 actions)
- [ ] Outputs are concrete deliverables (4-6 outputs)
- [ ] Boundaries clearly define scope
- [ ] Agent adds value beyond existing personas

## Integration with Commands

Agents created for workflows should be:
1. Referenced in parent command's personas list
2. Documented with generation timestamp
3. Linked to specific workflow patterns
4. Reusable across similar workflows
