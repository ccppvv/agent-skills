# Ally Orchestrator vs Dummy Skill - 职责划分

## 🎯 核心定位

### hi：跨领域任务分发中心

**职责：**
- 根据任务类型智能选择最合适的 ally
- 跨思考、执行、工作三大领域的任务编排
- 并行分发 + 集中监控 + 结果聚合

**范围：**
- 工作任务（bug修复、功能开发）→ tcodex-ally
- 思考任务（代码审查、架构设计）→ claude-ally
- 执行任务（脚本生成、配置文件）→ codex-ally
- 多模态任务（图像分析、UI对比）→ gemini-ally
- IDE集成任务（特定环境操作）→ codebuddy-ally（需显式指定）

### ok：工作任务专家

**职责：**
- 管理工作任务（jobs）的完整生命周期
- 状态机管理（N → I → P → A → C → D/W/T）
- 复杂度感知（0/1/2 三级）
- 依赖管理和优先级排序

**范围：**
- 仅处理工作类任务（bug修复、功能实现、代码重构）
- 细粒度的任务编排和状态跟踪
- worktree 隔离和并发控制

## 🔄 协作模式

### 模式 1：hi 直接分发

```
用户请求
    ↓
hi（智能路由）
    ↓
┌───┴────┬────────┬────────┬─────────┐
│ tcodex │ claude │ codex  │ gemini  │
└────────┴────────┴────────┴─────────┘
```

**适用场景：**
- 跨领域的混合任务批次
- 需要不同专家能力的任务
- 简单的并行任务执行

**示例：**
```json
{
  "tasks": [
    {"type": "work", "prompt": "修复登录bug"},      // → tcodex
    {"type": "think", "prompt": "审查代码安全性"},  // → claude
    {"type": "execute", "prompt": "生成配置文件"}   // → codex
  ]
}
```

### 模式 2：hi → tcodex → ok

```
用户请求
    ↓
hi（识别为工作任务批次）
    ↓
tcodex-ally
    ↓
ok skill（细粒度编排）
    ↓
┌────┬────┬────┬────┐
│job1│job2│job3│job4│（worktree隔离）
└────┴────┴────┴────┘
```

**适用场景：**
- 纯工作任务批次（全是 bug 修复或功能开发）
- 需要复杂状态管理的任务
- 需要依赖管理和优先级排序
- 需要 worktree 隔离的并行开发

**示例：**
```json
{
  "tasks": [
    {
      "type": "work",
      "prompt": "使用 ok 管理以下工作任务：[job列表]"
    }
  ]
}
```

### 模式 3：Codex Agent 内部使用 ok

```
用户在 Codex CLI 中
    ↓
直接使用 ok skill
    ↓
ok（完整生命周期管理）
    ↓
┌────┬────┬────┐
│job1│job2│job3│
└────┴────┴────┘
```

**适用场景：**
- 在 Codex/TCodeX CLI 会话中
- 需要完整的 job 管理功能
- 需要暂停/恢复/状态查询等高级功能

## 📊 功能对比

| 功能 | hi | ok |
|------|-------------------|-------|
| **核心职责** | 跨领域任务分发 | 工作任务专家 |
| **智能路由** | ✅ 根据类型自动选择 ally | ❌ 不负责路由 |
| **状态机** | 简单（4状态） | 完整（8状态） |
| **复杂度感知** | ❌ 无 | ✅ 0/1/2 三级 |
| **依赖管理** | ❌ 无 | ✅ 完整依赖图 |
| **优先级排序** | 简单（按顺序） | ✅ 智能算法 |
| **并发控制** | ❌ 无限制 | ✅ 5个/项目 |
| **Worktree隔离** | ❌ 无 | ✅ 完整支持 |
| **生命周期管理** | 基础 | 完整（pause/resume/kill） |
| **适用领域** | 跨领域混合 | 纯工作任务 |

## 🎨 使用建议

### 何时使用 hi

✅ **使用场景：**
- 需要跨领域协作（思考+执行+工作）
- 任务类型多样，需要不同专家
- 快速并行执行，无需复杂编排
- 简单的批次任务处理

❌ **不适用：**
- 纯工作任务，需要状态管理
- 需要依赖管理和优先级排序
- 需要 worktree 隔离的并行开发

### 何时使用 ok

✅ **使用场景：**
- 纯工作任务（bug修复、功能开发）
- 需要完整的生命周期管理
- 需要依赖管理和优先级排序
- 需要 worktree 隔离
- 需要暂停/恢复/状态查询

❌ **不适用：**
- 跨领域混合任务
- 需要思考/执行任务的场景
- 简单的一次性并行执行

## 🔧 实际案例

```json
{
  "batch_name": "auth-feature",
  "tasks": [
    {
      "type": "think",

    },
    {
      "type": "work",

    },
    {
      "type": "execute",

    },
    {
      "type": "think",
      "prompt": "审查代码安全性"
    }
  ]
}
```

### 案例 2：Bug 批量修复（推荐 ok）

在 tcodex-ally 中：
```bash
tcodex -- exec "使用 ok 修复以下 bugs"
```

ok 内部管理：
```markdown
| ID | Type | Title | Status | Complexity |
|----|------|-------|--------|------------|
| 1  | Bug  | 登录空指针 | N | 1 |
| 2  | Bug  | 数据库连接 | N | 2 |
| 3  | Bug  | API 超时 | N | 1 |
```

### 案例 3：混合工作流（组合使用）

```json
{

  "tasks": [
    {
      "type": "think",
      "ally": "claude",
      "prompt": "分析现有架构，给出重构方案"
    },
    {
      "type": "work",
      "ally": "tcodex",
      "prompt": "使用 ok 管理以下重构任务：[job列表]"

    {
      "type": "think",
      "ally": "claude",
      "prompt": "验证重构结果，审查代码质量"
    }
  ]
}
```

## 📝 总结

**简单记忆：**
- **hi** = **总控** → 根据任务类型选择专家
- **ok** = **工作专家** → 管理工作任务的完整生命周期

**协作关系：**
- hi 识别工作任务 → 分发给 tcodex-ally
- tcodex-ally 接收任务 → 可选使用 ok 进行细粒度编排
- Codex Agent 内部 → 直接使用 ok

**选择原则：**
1. 跨领域混合任务 → hi
2. 纯工作任务 + 需要状态管理 → ok
3. 简单并行执行 → hi
4. 复杂编排 + worktree 隔离 → ok
