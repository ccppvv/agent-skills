# 多 App 生命周期与运营手册

## 一、工程结构建议

- `apps/`：每个 App 一个独立入口工程。
- `packages/core/`：账号、网络、日志、配置。
- `packages/billing/`：统一订阅 SDK 与权益层。
- `packages/design-system/`：通用 UI 组件与主题。

目标：共享能力沉淀到 package，业务差异放在 app 层。

## 二、开发阶段

1. 统一脚手架：新 App 1 小时内可运行。
2. 统一质量门禁：Lint、测试、构建都进 CI。
3. 统一配置中心：环境变量、密钥、开关集中治理。

## 三、调试阶段

- 崩溃：接入统一 crash 平台，按版本/设备/场景聚合。
- 性能：首屏时间、卡顿率、内存峰值设阈值告警。
- 计费：关键埋点必须覆盖 `paywall_view -> purchase_start -> purchase_success -> entitlement_active`。

## 四、发布阶段

1. 每个 App 固定发布节奏（例如双周）。
2. 使用 TestFlight 分层灰度（内部 -> 小流量 -> 全量）。
3. 发布后 24 小时重点观察：崩溃率、支付成功率、退款异常。

## 五、维护阶段

- 版本支持策略：明确最小 iOS 版本与淘汰节奏。
- 技术债池：每迭代留 20% 容量处理稳定性与重构。
- 依赖升级窗口：月度统一升级，避免 App 间漂移。

## 六、运营阶段

核心指标：
- 获客：安装转化率、CAC。
- 变现：付费转化率、ARPPU、订阅续费率。
- 留存：D1/D7/D30。
- 健康度：崩溃率、ANR（如跨端）、差评率。

动作模板：
1. 每周一次增长复盘（渠道、价格、Paywall 文案实验）。
2. 每月一次订阅策略复盘（试用时长、价格阶梯、优惠活动）。
3. 每季度一次架构复盘（复用率、构建效率、故障复盘）。
