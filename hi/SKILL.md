# hi — 跨领域任务编排中心

**hi = 智能路由器**，分析任务后自主选择最优执行路径。

## 触发条件

当用户消息以 `hi` 开头（作为命令前缀，非打招呼），或涉及以下意图时触发：

| 触发词/模式 | 示例 |
|-------------|------|
| `hi ` / `hi,` / `hi，` | "hi 帮我分析这个" / "hi, 修复那个bug" |
| 任务分发 | "把这些任务分发给 ally" |
| 多任务并行 | "并行执行这几个任务" |
| ally 路由 | "用 claude 分析" / "用 codex 生成" |

## 执行路径决策

hi 收到任务后，按以下决策树选择执行方式：

```
任务到达
  ↓
是简单、单一、局部的任务？
  ├─ 是 → spawn subagent（轻量、快速、无额外依赖）
  └─ 否 ↓
      需要外部 CLI 能力（Claude Code / Codex / Gemini）？
        ├─ 是 → 调用对应 xxx-ally skill
        └─ 否 → spawn subagent
```

### 决策矩阵

| 条件 | 执行方式 | 原因 |
|------|----------|------|
| 单文件修改、简单查询、格式化 | **subagent** | 轻量，无需外部 CLI |
| 代码生成/脚本编写（不需要复杂推理） | **subagent** | 内置能力足够 |
| 多文件重构、复杂 bug 定位 | **tcodex-ally** | 需要外部 CLI 的深度代码理解 |
| 架构审查、设计评估、深度分析 | **claude-ally** | 需要外部 CLI 的强推理能力 |
| 大规模代码生成、配置批量创建 | **codex-ally** | 需要外部 CLI 的代码生成能力 |
| 视觉/UI 相关、截图分析 | **gemini-ally** | 需要外部 CLI 的多模态能力 |
| 用户显式指定 ally | **对应 ally** | 用户意图优先 |
| 3+ 个独立子任务 | **多 subagent 并行** | 并行效率最优 |

### subagent 选择

当决策为 spawn subagent 时，根据任务特征选择类型：

| 任务特征 | subagent_type | 说明 |
|----------|---------------|------|
| 需要代码搜索、文件查找 | `Explore` | 快速搜索，只读 |
| 需要规划、架构设计 | `Plan` | 探索+分析，只读 |
| 需要读写文件、执行命令 | `general-purpose` | 全能力 |
| 简单快速操作 | `general-purpose` + `lite` 模型 | 快速低成本 |

## Ally 路由表

| 任务类型 | Ally Skill | 关键词 |
|---------|-----------|--------|
| **工作任务** | `tcodex-ally` | 修复、实现、开发、bug、feature、重构 |
| **思考任务** | `claude-ally` | 审查、分析、设计、评估、优化、研究 |
| **执行任务** | `codex-ally` | 生成、创建、编写、脚本、配置 |
| **多模态任务** | `gemini-ally` | 截图、图片、视觉、UI、界面 |

## 工作流

```
用户 → "hi + 任务描述"
     ↓
   hi 分析任务类型 + 复杂度
     ↓
   ┌─────────────────────────────────┐
   │ 决策：ally 还是 subagent？       │
   ├────────────────┬────────────────┤
   ↓                ↓                ↓
 ally skill     subagent(1)     subagent(N)
 (Skill 工具)   (Agent 工具)    (Agent 工具并行)
   ↓                ↓                ↓
 bridge.py      直接执行        各自执行
 → CLI          → 返回结果      → 汇总返回
   ↓
 返回结果
     ↓
   hi 汇总返回用户
```

## 核心原则

1. **hi 是路由器，不是执行器** — 不直接使用 Bash/Read/Edit 完成任务
2. **能 spawn 就不 ally** — 简单任务用 subagent 更快更轻，减少外部依赖
3. **需要 CLI 才用 ally** — 只有当任务需要外部 CLI（Claude Code/Codex/Gemini）的专属能力时才调用 ally
4. **用户显式指定优先** — 用户说"用 claude"就走 claude-ally，不论任务复杂度
5. **多任务并行** — 独立任务同时派发，不串行等待

## 与 ok 的关系

- **hi**：跨领域总控，负责任务分类和智能分发（ally / subagent 双路径）
- **ok**：工作任务专家，管理 jobs.md 中 job 的生命周期
- 协作：hi 分发工作任务给 tcodex-ally 时，ally 内部可用 ok 细粒度编排
